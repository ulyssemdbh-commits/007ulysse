Ulysse

Très bon morceau, c’est déjà solide. Je te fais une review ciblée + quelques améliorations concrètes.

✅ Points forts
Parsing robuste des commandes [KANBAN_...] avec regex + normalisation FR/EN (priorité / statut).
Abstraction storage → logique métier découplée de la persistence.
Résultats uniformes (KanbanActionResult avec success, summary, data, error).
Recherche de tâche intelligente (findTaskByTitle avec exact + partial + gestion du multiple).
parseDateString qui gère langage naturel FR/EN (“aujourd’hui”, “demain”, “semaine prochaine”).
executeTaskSummary déjà exploitable pour un vrai briefing.
Maintenant, les vrais axes d’amélioration.

⚠️ 1. Incohérence avec ton schema DB / types
Tu utilises InsertTask de @shared/schema. Dans ce schema, aujourd’hui:

status: "todo" | "in_progress" | "done" (dans le schema que tu m’as montré plus haut)
Ici, côté Kanban, tu as:
'backlog' | 'todo' | 'in_progress' | 'review' | 'done'
Même chose pour priority:

DB: "low" | "medium" | "high" (a priori)
Ici: 'low' | 'medium' | 'high' | 'urgent'
Risque:

TypeScript compile (InsertTask peut être plus permissif selon ton zod),
mais tu risques de stocker des status="backlog" ou priority="urgent" que ton UI ne gère pas.
👉 À faire:

soit tu étends vraiment le schema Drizzle/Zod (ajouter backlog, review, urgent côté DB),
soit tu restreins Kanban pour coller au schema réel:
map backlog → "todo",
map review → "in_progress" ou autre,
map urgent → high + tag à part.
Tant que ce n’est pas aligné, tu as une bombe à retardement entre Kanban interne et UI Tasks.

⚠️ 2. projectId: projectId || null alors que Drizzle attend peut-être non-null
Dans:

const taskData: InsertTask & { userId: number } = {
  title,
  description: ...,
  projectId: projectId || null,
  priority: action.priority || 'medium',
  status: action.status || 'todo',
  dueDate: dueDate,
  userId,
};
Si projectId est undefined, tu forces à null.

OK si ton schema Drizzle déclare projectId nullable.
Sinon, tu vas te manger une erreur au moment de l’insert.
À vérifier, mais si projectId est NOT NULL en DB, il faut soit:

rendre projectId optionnel dans InsertTask,
soit imposer un projet “par défaut” pour les tâches Kanban.
✅ 3. Robustesse et UX de la recherche de tâches

Tu fais includes pour les partial matches:

  t.title.toLowerCase().includes(lowerTitle)

Si tu as beaucoup de tâches, c’est tout à fait OK (ça reste O(n)), mais:

✅ 4. Parsing et normalisation des commandes
Franchement, les regex sont bien foutues, mais deux améliorations possibles:

a) Autoriser les valeurs avec espaces dans titre / description
Tu as:

titre="Ma tâche"
description="Détails"
Mais ton pattern limite les valeurs à [^"\],]+ → pas de virgule, mais les espaces passent.
Ça marche pour beaucoup de cas, mais si tu veux autoriser des virgules dans la description:

soit tu acceptes des JSON-like,
soit tu restes comme ça (perso, pour un contrôle par IA interne, c’est largement suffisant).
Je ne toucherais pas tant que tu n’as pas un cas concret qui casse.

b) Gérer les ID de tâches dans les commandes
Tu le suggères dans les messages d’erreur:

“Utilise le titre exact ou l’ID.”

Mais tu n’as pas d’option [KANBAN_MODIFIER: id=123, ...] ni KANBAN_FAIT: id=....

👉 Amélioration simple:

étendre le pattern UPDATE / COMPLETE / DELETE pour accepter id=123,
et dans executeUpdateTask, si action.taskTitle ressemble à un ID numérique, l’utiliser direct.
Exemple minimal:

interface KanbanUpdateTaskAction {
  type: 'update_task';
  taskTitle?: string;
  taskId?: number;
  ...
}
adaptation des regex et de findTaskByTitle ou nouveau findTaskById.
⚠️ 5. Gestion du temps pour “en retard”
Dans executeTaskSummary:

const overdue = tasks.filter(t => {
  if (!t.dueDate || t.status === 'done') return false;
  return new Date(t.dueDate) < new Date();
});
Tu compares en incluant l’heure, alors que tu normalises à minuit dans parseDateString.
Pas dramatique, mais tu peux rendre ça cohérent:

const today = normalizeToMidnight(new Date());

  if (!t.dueDate || t.status === 'done') return false;
  const due = normalizeToMidnight(new Date(t.dueDate));
  return due < today;

Cela évite le “en retard” à 00h01 quand la date est aujourd’hui.
