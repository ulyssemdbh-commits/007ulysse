# DevOps Roadmap – Ulysseproject (Ligue des Champions DevOps)

## Objectif global

Mettre en place un **système DevOps intelligent, auto-apprenant et piloté par Ulysse**, capable de :

- **Voir** l’état réel du code et des dépendances (X-Ray),
- **Orchestrer** chaque changement (CI Oracle + Patch Advisor),
- **Apprendre** en continu à partir des échecs (Homework + Brain),
- **Corréler** incidents ↔ code ↔ historique DevOps,
- Offrir à Maurice un rôle de **coach** plutôt que de pompier.

Cette roadmap s’articule autour de **5 axes** complémentaires.

---

## Axe 1 – VISION & X-RAY (Observabilité du code)

### Objectif

Disposer d’une vue temps réel des :

- zones critiques du code (fichiers, domaines, modules),
- dépendances (imports, schéma, services),
- tendances de fragilité (improving / stable / degrading).

### Composants existants

- `devopsIntelligenceEngine.ts`
  - `BRAIN_IMPACT_MAP` : graphe de dépendances enrichi (fichier → domaines → imports/exports).
  - `DYNAMIC_FRAGILITY_LEARNING` (via `devops_file_history`) : fragilité dynamique par fichier.
  - `CI_ORACLE` : scoring de risque par changement.
  - `AUTO_PATCH_ADVISOR` : génération + ranking de patchs.
  - `HOMEWORK_BRAIN_PLANNER` : détection de gaps d’apprentissage.
  - `runFullDevOpsIntelligence()` : point d’entrée unifié.

### Actions à implémenter

1. **Exécution automatique de DevOps Intelligence**

   - [ ] Ajouter un service `devopsIntelligenceService` côté serveur qui :
     - récupère la liste des fichiers modifiés (via Git / devops_github),
     - construit l’`ImpactMap`,
     - calcule `CIRisk` et `PatchAdvice`,
     - met à jour `DYNAMIC_FRAGILITY_LEARNING` via `devops_file_history`,
     - persiste un résumé dans `Brain`.

   - [ ] Exposer une route API :
     - `GET /api/devops/intel/report?sha=...`
     - `POST /api/devops/intel/run` (avec payload: files/changes/bugType).

2. **Dashboard `/devops` – Vue X-Ray**

   - [ ] Ajouter un onglet “Intelligence” dans `/devops` qui affiche :
     - Top fichiers fragiles (via `/api/devops/fragility/leaderboard`),
     - Domaines analysés (auth, SUGU, sports, brain, etc.),
     - Derniers rapports `DevOpsIntelligenceReport`,
     - Tendances : improving/stable/degrading par domaine et par fichier.

### Critères de succès

- En 10 secondes, on peut répondre :
  - “Quels sont les 5 fichiers les plus fragiles aujourd’hui ?”
  - “Quels domaines se dégradent ?”
  - “Quel est l’impact prévisible d’un changement sur tel fichier ?”

## Axe 2 – EXÉCUTION ORCHESTRÉE (PR / CI / Merge)

### Objectif

Qu’aucun commit ou PR critique ne soit mergé sans :

- analyse de risque,
- visualisation de l’impact,
- stratégie de patch/rollback.

### Composants existants

- Bridge GitHub : `devops_github` (actions: `list_commits`, `list_prs`, `get_file`, etc.)
- `devopsIntelligenceEngine.runFullDevOpsIntelligence`
- `DYNAMIC_FRAGILITY_LEARNING` (`devops_file_history`)

### Actions à implémenter

1. **Hook PR → DevOps Intelligence**

   - [ ] Créer un endpoint serveur :

     - `POST /api/devops/intel/pr-analyze`
       - Input : `repo`, `prNumber`
       - Steps :
         - récupérer les diffs via `devops_github`,
         - construire `files` + `changes` pour DevOpsIntelligence,
         - appeler `runFullDevOpsIntelligence(...)`,
         - retourner : `ciRisk`, `patchAdvice`, `impactMap`, `learningGaps`.

   - [ ] Créer un GitHub Action / webhook :
     - Sur `pull_request` (opened / synchronize / reopened):
       - appelle `pr-analyze`,
       - poste un commentaire automatique sur la PR avec :
         - ⚠️ `riskLevel` (safe / caution / risky / dangerous),
         - fichiers fragiles touchés,
         - domaines impactés,
         - `bestChoice` du Patch Advisor,
         - recommandations.

2. **Règles de merge selon le risque**

   - [ ] Définir règles simples :
     - `riskLevel = safe` → merge OK.
     - `riskLevel = caution` → tests ciblés recommandés.
     - `riskLevel = risky` → review humaine obligatoire + plan de rollback.
     - `riskLevel = dangerous` → review + staging + rollback prêt.

   - [ ] Option : appliquer des labels GitHub automatiques :
     - `devops:risky`, `devops:dangerous`, `devops:needs-review`.

3. **Plan de rollback**

   - [ ] Générer un plan textuel simple à partir des diffs :
     - `git revert <sha>`,
     - endpoints/flows à surveiller après déploiement.

### Critères de succès

- Chaque PR importante arrive avec un **rapport DevOps** automatique.
- Plus de “merge à l’aveugle” sur des zones critiques.
