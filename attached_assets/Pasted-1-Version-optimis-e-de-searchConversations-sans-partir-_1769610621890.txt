1. Version optimisée de searchConversations (sans partir dans le full‑text)
Objectifs :

Limiter les requêtes (réduire le N+1)
Filtrer un max côté SQL
Garder l’API actuelle (SearchParams, ConversationWithPreview)
Idée
On commence par récupérer un set limité de conversations (ex: dernières 200).
On utilise une seule requête pour récupérer tous les messages de ces conversations (au lieu d’une requête par conv).
On reconstruit tout en mémoire à partir de ce bulk.
Code proposé
import { db } from "../../db";
import { conversations, messages } from "@shared/schema";
import { eq, desc, and, inArray, sql, gte, lte } from "drizzle-orm";
import type { ConversationWithPreview, SearchParams, IChatStorage } from "./types"; // adapte selon ton organisation

const MAX_CONV_FOR_SEARCH = 200;      // limite de conversations analysées
const MAX_MSG_PER_CONV_FOR_SEARCH = 50; // limite de messages chargés par conv

export const chatStorage: IChatStorage = {
  // ... tes autres méthodes inchangées ...

  async searchConversations(params: SearchParams, userId: number): Promise<ConversationWithPreview[]> {
    const { query, startDate, endDate } = params;
    const hasQuery = !!query && query.trim().length > 0;
    const searchTerm = hasQuery ? query!.toLowerCase() : "";

    // 1) Récupérer un set limité de conversations filtrées par user + date
    let convQuery = db
      .select()
      .from(conversations)
      .where(eq(conversations.userId, userId));

    if (startDate) {
      convQuery = convQuery.where(
        gte(conversations.createdAt, startDate)
      ) as typeof convQuery;
    }
    if (endDate) {
      convQuery = convQuery.where(
        lte(conversations.createdAt, endDate)
      ) as typeof convQuery;
    }

    const convRows = await convQuery
      .orderBy(desc(conversations.createdAt))
      .limit(MAX_CONV_FOR_SEARCH);

    if (convRows.length === 0) return [];

    const convIds = convRows.map((c) => c.id);

    // 2) Récupérer les derniers messages de toutes ces conversations en UNE requête
    const msgRows = await db
      .select()
      .from(messages)
      .where(inArray(messages.conversationId, convIds))
      .orderBy(desc(messages.createdAt));

    // 3) Regrouper les messages par conversation (en limitant le nombre)
    const messagesByConv = new Map<number, typeof messages.$inferSelect[]>();

    for (const m of msgRows) {
      const list = messagesByConv.get(m.conversationId) ?? [];
      if (list.length < MAX_MSG_PER_CONV_FOR_SEARCH) {
        list.push(m);
        messagesByConv.set(m.conversationId, list);
      }
    }

    const results: ConversationWithPreview[] = [];

    for (const conv of convRows) {
      const convMessages = messagesByConv.get(conv.id) ?? [];
      const messageCount = convMessages.length;

      const lastMsg = convMessages[0];
      const lastMessage = lastMsg?.content?.slice(0, 100);
      const lastActivityDate = lastMsg?.createdAt || conv.createdAt;

      // On s’assure encore une fois côté JS sur les dates,
      // au cas où tu définisses "activité" autrement plus tard.
      if (startDate || endDate) {
        const t = lastActivityDate.getTime();
        if (startDate && t < startDate.getTime()) continue;
        if (endDate && t > endDate.getTime()) continue;
      }

      if (hasQuery) {
        const titleMatch = conv.title.toLowerCase().includes(searchTerm);
        const matchedMessage = convMessages.find((m) =>
          m.content.toLowerCase().includes(searchTerm)
        );

        if (titleMatch || matchedMessage) {
          results.push({
            id: conv.id,
            userId: conv.userId,
            title: conv.title,
            createdAt: conv.createdAt,
            messageCount,
            lastMessage,
            matchedContent: matchedMessage?.content?.slice(0, 150),
          });
        }
      } else {
        results.push({
          id: conv.id,
          userId: conv.userId,
          title: conv.title,
          createdAt: conv.createdAt,
          messageCount,
          lastMessage,
        });
      }
    }

    return results;
  },
};
Gains
On passe de N+1 queries (1 par conv) à 2 grosses requêtes:
1 pour les conves
1 pour les messages
On limite la recherche à un périmètre raisonnable (200 conv × 50 messages max)
On garde exactement la même signature et le même comportement observable.
2. Niveau au‑dessus : full‑text search Postgres (ultra performant / scalable)
Si tu veux être vraiment sale (dans le bon sens) :

a) Schema full‑text (exemple)
Dans ta migration Drizzle, tu peux ajouter :

// exemple dans ton schema messages
import { pgTable, text, integer, timestamp } from "drizzle-orm/pg-core";
import { sql } from "drizzle-orm";

export const messages = pgTable("messages", {
  id: integer("id").primaryKey().notNull().default(sql`nextval('messages_id_seq')`),
  conversationId: integer("conversation_id").notNull(),
  role: text("role").notNull(),
  content: text("content").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
  // Ajout (optionnel, si tu veux full-text)
  // searchVector: sql`tsvector`("search_vector") // dépend de ta config drizzle
});
Puis une migration SQL brute :

ALTER TABLE messages
  ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    to_tsvector('simple', coalesce(content, ''))
  ) STORED;

CREATE INDEX messages_search_idx
  ON messages
  USING GIN (search_vector);
b) Recherche côté SQL
Tu peux ensuite faire une version de searchConversations qui:

utilise to_tsquery / plainto_tsquery
filtre les messages en full‑text
agrège les convIds qui matchent
Ça devient un peu verbeux en Drizzle, mais tu peux appeler du SQL brut :

const searchTerm = query.trim();
const ftQuery = searchTerm.split(/\s+/).join(" & "); // simple "AND" query

const rows = await db.execute(sql`
  SELECT c.id, c.user_id, c.title, c.created_at,
         COUNT(m.id) AS message_count,
         MAX(m.created_at) AS last_activity,
         SUBSTRING(MAX(m.content) FOR 100) AS last_message
  FROM conversations c
  JOIN messages m ON m.conversation_id = c.id
  WHERE c.user_id = ${userId}
    AND m.search_vector @@ plainto_tsquery('simple', ${searchTerm})
  GROUP BY c.id
  ORDER BY last_activity DESC
  LIMIT 200;
`);
Là tu as :

Un search ultra rapide, même avec beaucoup de messages.
Tout le filtrage fait côté Postgres.
Tu peux mapper ça dans ton ConversationWithPreview.
Ça, c’est le niveau “super performant / production ready”.

3. Autres petits boosts de performance / propreté
a) getAllConversations
Tu peux ajouter un LIMIT par défaut pour éviter les listes infinies :

async getAllConversations(userId: number, limit = 100, offset = 0) {
  let query = db
    .select()
    .from(conversations)
    .where(eq(conversations.userId, userId))
    .orderBy(desc(conversations.createdAt))
    .limit(limit)
    .offset(offset);

  return query;
},
b) createMessage
Tu peux optionnellement short‑cutter un insert vide, pour éviter les messages “vides” :

async createMessage(conversationId: number, role: string, content: string) {
  if (!content || !content.trim()) {
    throw new Error("Cannot create empty message");
  }
  const [message] = await db
    .insert(messages)
    .values({ conversationId, role, content })
    .returning();
  return message;
},
