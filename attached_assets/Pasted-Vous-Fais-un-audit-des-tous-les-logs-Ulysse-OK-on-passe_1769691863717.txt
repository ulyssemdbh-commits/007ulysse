2️⃣ Ce que je regarderais dans les logs (audit analytique)

Une fois qu’on a tout ça, voilà l’audit que je ferais:

A. Performance / Latence
Questions:

Latence moyenne globale MARS ?
95ᵉ percentile ? (combien de requêtes “lentes” > X secondes)
Combien de requêtes utilisent réellement > 10s du budget 15s ?
Objectifs:

Si 80–90 % des requêtes < 5–6s → on peut éventuellement réduire le budget ou optimiser encore.
Identifier:
requêtes ultra lentes (profil particulier ? type “site JS lourd” ?)
domaines souvent crawlés en Playwright → candidats à un profil Jina/Playwright dédié.
B. Usage Perplexity vs Serper
Questions:

% de requêtes où Perplexity est réellement utilisé ?
% de timeouts Perplexity ?
% de circuit-breaker activé ?
Comparaison:
temps moyen Serper vs Perplexity
amélioration de confidenceLevel quand Perplexity est présent ou non
Objectifs:

Décider:
si Perplexity doit être activé pour tous les types de requêtes ou seulement certains (actu, explications complexes…)
si le timeout peut être réduit/incrémenté intelligemment
C. Efficacité de l’early-exit
Questions:

% de requêtes où wasEarlyExit = true
Motifs:
high_confidence_answer
time_budget_exhausted
no_better_sources
Comparer la latence:
early-exit vs full pipeline
Objectif:

Vérifier que l’early-exit sert vraiment à gagner du temps sur les cas faciles, et pas à couper trop tôt sur les cas complexes.
D. Qualité / Politique anti-approx
Questions:

% de requêtes avec canRespond = false (refus)
Typologie de ces refus:
live_score / cotes / prix temps réel
questions trop vagues
sources insuffisantes
Voir si:
des requêtes simples sont refusées à tort (policy trop stricte)
des types peuvent être traités avec une logique spécifique (par ex, renvoyer vers une page référente)
E. Spécifique sports / Djedou Pronos
Questions:

Quand une question contient: “match, cote, prono, Ligue 1, Champions League…” :
sportsContextInjected = true ?
matchesFromCache > 0 ?
Latence:
temps MARS vs temps purement cache sports
% de cas où les cotes sont absentes (cache pas à jour / API KO)
Objectifs:

Vérifier que:
Ulysse utilise bien le cache pour les matchs du jour (pas de call externe inutile)
les données injectées contiennent systématiquement cotes + tags (SAFE, VALUE, etc.)
le système de prédiction multi-sports est bien exploité
3️⃣ Plan d’optimisation après audit (ce que je ferai avec les résultats)

En fonction de ce que les logs montreraient, voilà les actions typiques:

Si latence moyenne > 6–7s

Réduire le recours systématique à Perplexity pour certains types de requêtes
Augmenter le poids du cache / de l’historique MARS pour les requêtes récurrentes
Créer un profil smartCrawl par domaine problématique (Twitter-like, sites JS lourds, etc.)
Si Perplexity timeouts/circuit-breaker explosent

Diminuer le timeout
Le déclencher uniquement quand Serper + smartCrawl n’atteignent pas un certain confidenceLevel
Si beaucoup de refus canRespond=false sur des choses simples

Ajuster légèrement les seuils de confidence ou les critères pour certains types (définitions, généralités)
Ajouter plus de domaines “trusted” prédéfinis avec haut score (Wikipedia, docs officielles, etc.)
Si sports cache sous-exploité

Renforcer la détection sémantique des requêtes sports
Toujours tenter d’utiliser le cache AVANT MARS pour les datas du jour (matchs, cotes)
Monitorer cache staleness (date dernière sync, etc.)
