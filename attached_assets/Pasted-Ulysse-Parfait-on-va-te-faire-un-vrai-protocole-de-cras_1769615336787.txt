2. Comment tester concrètement
2.1. Test de toutes les stratégies sur une URL
Pour chaque URL ci‑dessus:

Appel /scrape/smart/test:
curl "http://localhost:PORT/api/v2/scrape/smart/test?url=URL_A_TESTER"
ou via Postman / ton front.

Regarde la réponse:
Tu vas avoir quelque chose comme:

{
  "success": true,
  "url": "https://www.parionssport.fdj.fr/paris-football/ligue-1",
  "results": [
    {
      "strategy": "http",
      "success": false,
      "contentLength": 120,
      "qualityScore": 0.18,
      "durationMs": 450,
      "preview": "...",
      "error": null
    },
    {
      "strategy": "playwright",
      "success": true,
      "contentLength": 5200,
      "qualityScore": 0.72,
      "durationMs": 3800,
      "preview": "OM - PSG 2.15 3.40 2.90...",
      "error": null
    },
    {
      "strategy": "jina",
      "success": true,
      "contentLength": 4300,
      "qualityScore": 0.68,
      "durationMs": 1200,
      "preview": "...",
      "error": null
    },
    ...

Ce qu’il faut regarder par URL:
strategy → qui marche vraiment sur ce domaine ?
success → true/false (selon ton minQuality dans smartCrawl)
contentLength / preview → tu vois si c’est du vrai contenu ou du bruit
qualityScore → comment ça se situe:
< 0.3 → extraction douteuse
0.3–0.6 → moyen
0.6 → correct/bon

durationMs → coût per stratégie

3. Comment interpréter pour ajuster ton moteur
Voici ce qu’on va chercher:

A. Sur les bookmakers (ParionsSport, Winamax, Betclic)
Sur /scrape/smart/test :

Si http:
success: false ou qualityScore < 0.3 → normal
Si playwright:
success: true, qualityScore > 0.5, contentLength > ~1500 → 👍
Si jina / firecrawl:
Si l’un d’eux donne du contenu correct en moins de temps que playwright, c’est un super signal.
Ce qu’on en déduit:

Si playwright est toujours le meilleur → jsRequired = true, lastSuccessStrategy = "playwright" → parfait.
Si jina est souvent aussi bon mais plus rapide → on pourra un jour le mettre en priorité plus haute (priority) pour certains domaines.
Tu peux ensuite appeler /scrape/smart/health et /scrape/smart/analytics pour voir:

le succès global de playwright vs http sur ces domaines,
comment ta moyenne de qualité évolue.
B. Sur les articles (L’Équipe, Le Monde, blog)
Sur /scrape/smart/test:

http doit:
success: true
qualityScore > 0.5
contentLength bien fourni
playwright peut être plus lent sans apporter grand-chose → normal.
Là, ton engine doit retenir http comme stratégie préférée:

jsRequired devrait rester false
lastSuccessStrategy → "http"
successCount augmenté
Si tu vois que jina donne aussi un bon résultat:

ça reste un bon fallback si un jour http se fait bloquer.
C. Sur les sites SPA (Next docs, Nuxt docs)
Sur /scrape/smart/test:

http:
souvent qualityScore moyen (un peu de texte mais pas tout)
playwright:
devrait monter le score (plus de texte)
metrics.jsDetected + jsRequired → doivent être true après quelques appels.
Si tu vois que malgré playwright, qualityScore reste faible, on ajustera soit:

la logique de détection SPA (detectSPAIndicators),
soit la configuration (timeout, etc.).
D. Sur PDF via Firecrawl / Jina
Sur /scrape/smart/test:

http → success: false, normal.
firecrawl / jina:
si l’un te donne le texte du PDF → victoire.
Si c’est le cas, on pourra:

augmenter leur priority pour les URLs qui contiennent .pdf ou certains patterns de domaine (plus tard).
