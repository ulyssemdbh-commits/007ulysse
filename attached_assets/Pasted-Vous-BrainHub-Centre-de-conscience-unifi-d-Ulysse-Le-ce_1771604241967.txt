2. Gaps par rapport à tout ce qu’on a activé autour (KPI, auto-learning, décision, owner profile)
Vu tout ce qu’on a mis en place (KPI system, Decision Coach, Owner Profile 2.0, Auto-Learning Engine, Initiative Engine, etc.), il manque 4 intégrations majeures dans ce BrainHub:

2.1. Aucune connexion directe aux KPIs / auto-learning
Tu gères stats internes (inputs, outputs, actions, decisions, workingMemorySize…) mais:

Pas de:
latence par requête,
quality/factual flags,
satisfaction,
health score.
Pas de remontée vers /api/v2/ulysse/kpis.
👉 À faire pour coller au nouveau système:

Ajouter des hooks dans processInput (et éventuellement dans les callbacks de CoreEngine) pour:
loguer:
startTime, endTime, latency,
domaines (SUGU, sports, dev, perso…),
tools utilisés (hearing, vision, action, outils métier),
décision prise.
appeler un service type kpiTracker.record({ ... }).
Tu peux rajouter un champ dans BrainStats genre:

interface BrainStats {
  // ...
  lastLatencyMs?: number;
  avgLatencyMs?: number;
}
et un petit service dédié qui pousse vers /api/v2/ulysse/kpis.

2.2. Le Decision Coach et l’Owner Profile ne sont pas encore branchés ici
Ton makeDecision ne regarde que:

la charge cognitive,
la présence d’input récent,
l’urgence.
Mais maintenant, on a :

un Owner Profile Maurice 2.0 (paris, SUGU, dev),
un Decision Coach,
un Initiative Engine.
👉 Propositions concrètes:

Brancher le domain / intent

Ajouter au context de makeDecision un champ:
domain?: 'sugu' | 'sports' | 'dev' | 'personal' | 'generic';
Ce domain serait détecté en amont (par ton intent classifier actuel).
Décisions enrichies selon domaine

Pour domain === 'sports' + OwnerProfile → Decision Coach:
décide si on propose un pari,
ou si on dit “rien d’assez clean aujourd’hui”.
Pour domain === 'sugu' → Initiative Engine:
déclenche ou non une analyse d’anomalies / synthèse.
Faire remonter ça dans BrainDecision

Ajouter par ex:
interface BrainDecision {
  // ...
  domain?: string;
  strategy?: string; // ex: 'value_bet_pipeline', 'sugu_cashflow_analysis'
}
Et processInput pourrait appeler un decisionCoach externe après makeDecision, pour les cas “query de décision”, et enrichir la décision.

2.3. L’Initiative Engine (anomalies SUGU, briefs) n’est pas piloté ici
Ton BrainHub a déjà des background processes (décroissance de charge, cleanup mémoire), mais:

Il ne lance pas:
checks SUGU matin,
revue Monday,
proactivité contextuelle.
👉 Idée simple:

Ajouter dans startBackgroundProcesses() un hook vers un initiativeEngine:
private startBackgroundProcesses(): void {
  // existant...

  // Initiative checks

    initiativeEngine.tick(this.consciousness, this.stats);

2.4. L’Enhanced Self-Critique n’est pas branché dans le cycle
Ton BrainHub finit processInput en renvoyant une structure neutre:

return {
  response: '',
  shouldSpeak: ...,
  actions: [],
  decision,
};
Mais maintenant on a:

Self-critique post-réponse (qualité, factualité, alignement objectifs),

👉 Pattern à adopter:

BrainHub ne fera pas la self-critique lui-même, mais:
il peut:

taguer certains outputs (via processVoiceOutput) comme “à évaluer”.
Ensuite, un qualityEngine/selfCritiqueEngine est appelé par le CoreEngine après réponse, et:
ajoute une thought:
“Réponse X évaluée: OK / à améliorer pour telle raison”,
ou logue vers /api/v2/ulysse/kpis/error en cas d’erreur factuelle.
