3. handleSendMessage = entonnoir monstrueux
Symptôme
Cette fonction fait TOUT :

gère l’input texte,
ajoute le contexte fichier (pendingFileAnalysis),
envoie la requête HTTP,
gère le streaming SSE,
reconstruit fullResponse,

Ce que ça peut causer

Toute régression sur un aspect (fichiers, navigation, display, TTS) peut casser le flux global.
Difficile de tester une seule responsabilité (ex : “est-ce que la logique de display de fichier marche bien ?”).
Ce que je ferais

Le découper en petits morceaux pures :

buildOutgoingMessage(content, pendingFileAnalysis)
→ renvoie { content, imageDataUrl, clearedPending }

streamAssistantResponse(conversationId, payload)
→ renvoie fullResponse ou throw une erreur.
(juste le fetch + stream + reconstruction du texte)

applyAssistantResponseToCache(fullResponse)
→ ajoute le message assistant dans React Query.

runResponseSideEffects(fullResponse)
→ parse markers / navigation / display :

[AFFICHER_IMAGE: ...]
[AFFICHER_TEXTE: ...]
[NAVIGATION: ...]
speakAssistantResponse(fullResponse)
→ clean + speak.

Tu gardes handleSendMessage comme orchestrateur :

const handleSendMessage = useCallback(async (messageText?: string) => {
  const { content, imageDataUrl, clearedPending } = buildOutgoingMessage(...);
  // … post, stream, get fullResponse
  applyAssistantResponseToCache(fullResponse);
  await runResponseSideEffects(fullResponse);
  speakAssistantResponse(fullResponse);
}, [...]);
Ça réduit la complexité cognitive d’un gros bloc de 150+ lignes en 5 fonctions claires.
